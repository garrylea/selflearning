## 1. 引言
本文档在原有需求基础上，引入 **“Agentic Shell”** 核心设计思想。系统不再依赖死板的固定接口，而是通过赋予后端 Agent 强大的本地 Shell 工具链调用权限，实现自适应练习、智能搜题、多媒体对话的高动态闭环。

## 2. Agent 驱动的混合架构分析
系统分为三个核心层级：
*   **Knowledge Repository (静态库)**: 存储于本地磁盘的 Markdown 结构化题目集（由 Python 管道产出）。
*   **Agent Control Center (大脑)**: 
    *   **Logic Engine**: 处理 3/3 过关等核心算法。
    *   **Shell Executor**: Agent 通过执行 `rg`, `sed`, `grep` 等指令直接操作题目库。
*   **Visual Interface (感官)**: Tauri 驱动的图形化界面，负责展现知识图谱与沉浸式练习环境。

## 3. 核心功能模块拆解 (业务流程对齐)

### 3.1 预处理与静态展示 (无需 Agent)
- **试卷分析与入库**: 离线完成。Python 脚本将 PDF 转化为 Markdown，并在数据库中建立 `试卷ID-题号-路径` 的元数据索引。
- **科目与知识图谱选择**: 用户在 Tauri 前端选择年级、学期、科目。系统从数据库读取预设的知识树（思维导图格式）并渲染，无需 AI 参与。

### 3.2 自适应题目推送 (队列机制)
- **初始化**: 用户选中知识点后，系统查询其当前水平（A/B/C/D）。
    - **A 级处理**: 弹出提示框告知已掌握，询问是否重学，并允许手动选择 B/C/D 级别题目。
- **经典真题队列**: 系统从数据库检索该级别的 **3 道经典真题**，形成待做队列。
- **题目提取**: App 根据数据库中的路径，直接调用 `read_question` 工具从物理文件中提取 Markdown 内容展示给学生。

### 3.3 Agent 核心调用场景 (核心介入点)
Agent 及其工具链仅在以下场景启动：
1.  **错题/求助响应**: 学生做错或点击“不会做”时，Agent 介入，读取题目内容并生成**分步骤详细题解**。
2.  **变式强化 (The Variation Loop)**: 
    - 在学生确认看懂题解后，Agent 从题库中检索（或实时生成）一道**同类型变式题**。
    - **逻辑锁定**: 变式题未做对前，原 3 题队列挂起。变式题做对后，恢复原队列。
3.  **多轮互动讨论**: 在题解界面，学生针对该题细节发起的任何提问，均由 Agent 结合上下文回答，过程中 Agent 可按需调用工具查找相关例图或公式。
4.  **状态恢复 (Resume Learning)**: 当用户重新打开 App 时，Agent 读取 Session 日志，恢复之前的题目队列、当前做题位置及讨论历史。

## 4. 业务数据流与状态机分析
1.  **Start**: 用户选中知识点 -> 系统初始化 3 题队列 -> 推送第 1 题。
2.  **Correct**: 做对 -> 推送队列下一题。
3.  **Incorrect/Help (Trigger Agent)**: 
    - Agent 生成题解 -> 学生确认 -> Agent 推送变式题。
    - **Loop**: 变式题错误 -> 重新生成题解 -> 再次推送变式题。
    - **Success**: 变式题正确 -> 恢复原队列。
4.  **Finish**: 队列清空 -> 评估掌握度 -> 更新数据库。
5.  **Interrupt**: 用户中途退出 -> 记录 Session -> 下次启动时恢复。

## 5. 数据模型初步设计
- **`Questions_Index`**: 索引 Markdown 文件路径。
- **`Knowledge_Graph`**: 存储由 AI 自动推导的知识点前置依赖（有向无环图）。
- **`User_State`**: 存储用户掌握度 Level。
- **`Multimedia_Logs`**: 存储对话历史的音视频文件映射。

## 6. 技术可行性与风险评估
- **一致性风险**: 不同模型定级标准可能不一。采用“三轮投票定级法”缓解。
- **无效题风险**: 在 Python 管道中增加“AI 模拟解题”步骤，自动剔除逻辑不通或描述不清的题目。
- **性能**: 在局域网环境下，`ripgrep` 的搜索速度完全能支撑万级并发。
